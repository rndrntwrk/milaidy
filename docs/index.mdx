**Milady** is a client/server AI agent app built on [elizaOS](https://github.com/elizaOS).
The Milady client can create a local server on your machine, discover a server
on your LAN, connect to an existing remote gateway, or attach to Eliza Cloud.
Once connected, the selected server owns provider routing, tools, memory, and
runtime state through the Gateway control plane.

This Mintlify site documents the developer and operator side of Milady:
architecture, configuration, APIs, plugins, and runtime behavior. Developer
docs live at [docs.milady.ai](https://docs.milady.ai). Consumer-facing docs
live at [milady.ai/docs](https://milady.ai/docs).

Milady supports local inference and third-party model providers, has a growing
plugin ecosystem powered by elizaOS, a skills marketplace via
[ClawHub](https://clawhub.ai), and runs on every major platform: desktop,
mobile, web, CLI, and Chrome extension.

## Why Milady

### Client/Server by Default

Milady can create a local server on your own hardware, discover a server on
your LAN, or connect to a trusted remote or Eliza Cloud server without
changing the client model. The selected server owns provider routing,
conversations, tools, and runtime state.

### Plugin Ecosystem

Milady ships with 25+ first-party elizaOS plugins covering knowledge management, browser automation, computer use, Discord, Telegram, TTS (Edge TTS and ElevenLabs), PDF processing, shell access, cron scheduling, trust scoring, and more. Third-party plugins install with a single command through the plugin manager.

### Skills Marketplace

Skills are markdown-based extensions that teach your agent new capabilities without writing code. Browse and install skills from [ClawHub](https://clawhub.ai), the default skills registry, or point to your own registry.

### Autonomous Mode

Milady can operate autonomously -- executing scheduled tasks, running multi-step workflows, and maintaining its own loop of thoughts and actions. Configure triggers (interval, cron, or one-shot) and let the agent work on your behalf.

### Built-In Crypto Wallet

The onboarding flow can generate or import EVM and Solana wallets. Keys are stored locally in your config and available to plugins at runtime for on-chain interactions, trading, and DeFi.

## Architecture at a Glance

Milady is structured as a layered system:

```
 User Interfaces (Desktop / Mobile / Web / CLI)
                         |
                    API Server (HTTP + WebSocket on :2138)
                         |
                   Gateway (:18789)
                         |
                  elizaOS Runtime
                    |         |
            Milady Plugin    Other Plugins
            (personality,     (knowledge, browser,
             wallet, skills,   TTS, Discord, Telegram,
             autonomous state) shell, cron, ...)
                    |
              Connector Layer
         (Discord, Telegram, Slack, WhatsApp, Twitter,
          Signal, Farcaster, WeChat, Matrix, Nostr, ...)
```

- **elizaOS Runtime** -- the core agent framework that handles memory, message processing, action dispatch, and provider abstraction.
- **Milady Plugin** -- a first-party elizaOS plugin that provides workspace context, session keys, personality presets, autonomous state, and wallet integration.
- **Gateway** -- a WebSocket + HTTP multiplexed server that bridges the runtime to all connected clients and connectors.
- **Plugin System** -- standard elizaOS plugins that add capabilities (knowledge, browser, TTS, etc.).
- **Connector Layer** -- platform-specific adapters for Telegram, Discord, Slack, WhatsApp, Signal, iMessage, Blooio, MS Teams, Google Chat, Twitter, Farcaster, Twitch, Mattermost, WeChat, Matrix, Feishu/Lark, Nostr, Lens, and the web chat UI.

The runtime's primary persisted config lives in `~/.milady/milady.json`.
Canonical runtime routing is split across `deploymentTarget`,
`linkedAccounts`, and `serviceRouting`. Secrets can also come from
`~/.milady/.env`.
