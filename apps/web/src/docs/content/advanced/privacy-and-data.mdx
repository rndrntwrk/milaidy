# Privacy, data, and what stays local

Milady is local-first by design, but "local-first" isn't "nothing ever leaves your machine." If you use a cloud language model, your messages get sent to it. If you connect to Discord, your messages travel through Discord's servers. This page tells you exactly what happens to your data and how to tune what leaves your machine.

**What you'll learn:** where each piece of your data lives, what gets sent where during normal use, and how to minimize what leaves your machine if you care about that.

## What lives on your machine

Everything below lives in `~/.milady/` (macOS/Linux) or `%USERPROFILE%\.milady\` (Windows):

| What | Where | Notes |
| --- | --- | --- |
| **Configuration** | `milady.json` | Every setting from the UI. Including API keys (encrypted at rest). |
| **Conversations** | Local database | Every message you've ever sent and every response received. |
| **Characters** | `characters/` | Your character definitions — name, personality, voice preferences. |
| **Uploaded documents (knowledge)** | Local database | The original files plus their vector embeddings. |
| **Avatars (VRM files)** | `avatars/` or referenced from wherever you uploaded them. |
| **Logs** | `logs/` | Diagnostic logs. Rotated automatically. |
| **Cached data** | `cache/` | Model response cache, asset cache. Cleared on demand. |

This directory is **your data**. Back it up if you care. Migrate it to a new machine by copying it. Never delete it in anger.

<Callout kind="tip">
If you want to know what's in your Milady install without opening the app, open <code>~/.milady/</code> in a file browser. The directory structure is human-readable. The only thing you can't inspect directly is the SQLite conversation database (use a SQLite browser if you really want to).
</Callout>

## What leaves your machine during normal use

Here's the honest accounting. "Leaves your machine" means "gets sent over the network to another party."

### 1. Language model inference (chat provider)

**Every message you send is sent to your chat provider.** Plus context: the character's personality, recent conversation history, relevant knowledge chunks if any.

- If your provider is **local (Ollama)**: this stays on your machine. Nothing leaves.
- If your provider is **cloud (OpenAI, Anthropic, OpenRouter, Eliza Cloud, etc.)**: the request goes to their servers. They see the full context of your conversation during inference. Most providers retain logs for some period — read their privacy policy.

**What you can do about it:** use a local provider. Ollama + llama3.1 is a fully local setup where chat never leaves your machine. Tradeoff: slower and less capable than cloud models.

### 2. Voice synthesis (TTS)

Every response your agent speaks aloud gets sent to your TTS provider to synthesize.

- **Local TTS**: none of the big players offer a great local option yet. Edge TTS goes over the network, but to Microsoft's edge servers, which is faster than most US cloud.
- **Cloud TTS**: ElevenLabs, OpenAI, Google, etc. They see the text your agent is speaking. Most retain logs.

**What you can do about it:** turn voice off entirely if you don't need it. Text-only chat doesn't touch a TTS provider.

### 3. Speech recognition (STT)

If you use talk mode, your microphone audio gets sent to an STT provider for transcription.

- Some providers support on-device models (Whisper.cpp for example). If your STT is local, this stays on your machine.
- Cloud providers see your voice audio.

**What you can do about it:** use a local STT like Whisper, or don't use talk mode.

### 4. Embeddings for knowledge/memory search

When you upload a document or search your memory, Milady computes vector embeddings for the text. If your embeddings provider is cloud-based, the text gets sent there briefly during the embedding computation.

**What you can do about it:** use a local embeddings model (Ollama has `nomic-embed-text` and others that work well).

### 5. Connector traffic

If you've enabled Discord, Telegram, iMessage, etc., every message your agent receives or sends goes through that platform's servers. Discord sees all the Discord messages. Telegram sees all the Telegram messages. That's inherent to the platform — nothing Milady can do about it.

**What you can do about it:** don't connect to platforms you don't want sending messages through.

### 6. Telemetry (off by default)

Milady has opt-in telemetry that reports anonymized diagnostic data: crash reports, feature usage counts, performance metrics. **It is off by default on desktop.** You can enable it in Settings → Privacy → Telemetry if you want to help improve Milady.

Telemetry never includes conversation contents, API keys, character details, or anything personally identifiable. It's the kind of thing that tells the team "40% of users enable Discord" not "user X sent message Y."

### 7. Plugin behavior

Every plugin you install can make its own network requests. A web search plugin sends your queries to a search provider. A weather plugin talks to a weather API. A calendar plugin talks to your calendar provider. Plugins are listed in Settings → Plugins; each one documents what it sends where.

## What does NOT leave your machine

- Your character files and personality prompts (except when they're sent as context to your chat provider during inference — but the files themselves stay local).
- Your conversation history — it's stored locally. The provider sees individual messages during inference but never has a persistent copy of your full history.
- Your `milady.json` config — never transmitted anywhere.
- Your local database.
- Your `~/.milady/` directory as a whole.
- Your wallet private keys (if you use the wallet features) — stored encrypted at rest, used for signing locally, only the signed transactions go out on-chain.

## The "fully local" setup

If you want to minimize what leaves your machine, here's the stack:

| Slot | Choice |
| --- | --- |
| Chat provider | Ollama + a local model (llama3.1, qwen2.5, etc.) |
| TTS | Edge TTS (goes to MS servers but no auth / no account needed), or skip voice |
| STT | Local Whisper (via Ollama or a Whisper plugin), or skip talk mode |
| Embeddings | Ollama nomic-embed-text |
| Connectors | None, or only ones you explicitly want |
| Telemetry | Off (default) |
| Plugins | Only ones whose network behavior you're OK with |

With this setup, running a normal text-only chat conversation stays entirely on your machine. The only thing that touches the network is your model downloads when you first pull them from Ollama.

## Developer debug features

There are a few env vars that control developer diagnostic features. You should not turn any of these on unless you're actively debugging something:

- **`MILADY_CAPTURE_PROMPTS=1`** — dumps every raw prompt (including user messages) to disk under `.tmp/prompt-captures/`. **Never enable this in production use.** The capture files contain your conversations in plain text.
- **`MILADY_TTS_DEBUG=1`** — verbose TTS pipeline tracing. Safe to run, but noisy.
- **`MILADY_PROMPT_TRACE=1`** — logs prompt compaction stats to the console. Safe.

If you accidentally turned one of these on, delete the `.tmp/` directory in your Milady install and unset the env var.

## "How do I see what's happening in real time?"

Settings → Advanced → Observability shows you:

- Live logs from the runtime.
- A request log for every call Milady is making to external services (which provider, which endpoint, how long it took).
- Token usage counters for your language model provider.

This is the honest window into what Milady is doing behind the scenes. If you're worried about an unexpected network call, this is where you confirm what's actually happening.
