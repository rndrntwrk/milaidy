# Connect to iMessage (macOS)

Milady can send and receive real iMessages on a Mac that's signed in to your Apple ID. This is a **macOS-only** connector — it uses Messages.app directly through AppleScript, with an optional CLI backend for higher-throughput cases.

**Who this is for:** anyone running Milady on a Mac that's also signed in to iMessage. This is the whole setup: no bridge, no external server, no third-party credentials.

## What you need before you start

- **A Mac running macOS 11 or newer**, signed in to iMessage. Confirm by opening Messages.app and sending a message to yourself — if it's a blue bubble, you're good.
- **Milady installed and running on that same Mac.** This connector will not work on Windows or Linux because iMessage doesn't exist on those platforms.
- **Admin access** on the Mac to grant a couple of system permissions.

## Step 1 — Grant Full Disk Access and Accessibility

macOS blocks apps from reading iMessage data or controlling Messages.app by default. You have to explicitly allow both.

<Steps>
  <li>Open <strong>System Settings → Privacy &amp; Security → Full Disk Access</strong>.</li>
  <li>Click the <strong>+</strong> button and add <strong>Milady.app</strong>. Toggle it on.</li>
  <li>In the same Privacy &amp; Security panel, click <strong>Accessibility</strong> in the sidebar.</li>
  <li>Add <strong>Milady.app</strong> here too, and toggle it on. Accessibility is what lets Milady tell Messages.app to send replies on your behalf.</li>
  <li><strong>Quit Milady completely and relaunch.</strong> Both permissions only take effect for newly-launched processes.</li>
</Steps>

<Callout kind="warning" title="Full Disk Access is powerful">
Full Disk Access lets an app read every file the user account can see — not just iMessage data. Only grant it to software you trust, and check periodically what else has it.
</Callout>

## Step 2 — Configure the plugin

<Steps>
  <li>Open Milady. Go to <strong>Settings → Plugins → iMessage → Configure</strong>.</li>
  <li><strong>Leave CLI path and DB path blank.</strong> The plugin will use AppleScript against Messages.app, which works on any Mac with iMessage signed in.</li>
  <li>(Optional) Set <strong>Poll interval (ms)</strong> — how often Milady checks for new messages. Default 5000ms is fine for most uses; lower it for snappier responses.</li>
  <li>Click <strong>Save</strong>.</li>
</Steps>

## Step 3 — Test it

From your iPhone (or any other iMessage device on a different Apple ID), send a message to the Apple ID that's signed in on this Mac.

Within a few seconds — bounded by your poll interval — Milady should see the message and reply in the same thread. The reply will be a real blue bubble sent through Messages.app.

## Useful options

| Option | What it does |
| --- | --- |
| **DM policy** | `allow-all` responds to everyone. `allow-from` restricts replies to a list of contacts. |
| **Group policy** | `allow-all`, `allow-from`, or `deny-all` for group chats. |
| **Allow from** | Comma-separated phone numbers / emails allowed when policy is `allow-from`. Use E.164 format (`+15551234567`). |

## Advanced: the optional CLI backend

If the AppleScript path isn't fast enough for your use case (e.g. you need sub-second polling for hundreds of active conversations), the plugin can also use an external CLI called `imsg`. This is optional — the default AppleScript backend works fine for normal use.

To use it:

<Steps>
  <li>Install the <code>imsg</code> CLI (check the plugin's upstream repository for the current installer). Confirm it's on your <code>$PATH</code> by running <code>which imsg</code> in a terminal.</li>
  <li>In Milady's iMessage Configure panel, set <strong>CLI path</strong> to the absolute path printed by <code>which imsg</code>.</li>
  <li>Optionally set <strong>DB path</strong> to your chat.db location — defaults to <code>~/Library/Messages/chat.db</code> and rarely needs changing.</li>
  <li>Save and restart.</li>
</Steps>

If the CLI isn't found at the path you provide, the plugin silently falls back to AppleScript and logs a warning. You don't lose functionality either way.

## Troubleshooting

**"Permission denied" reading iMessage data.**
Full Disk Access wasn't granted, or it was granted before Milady was launched. Re-check System Settings → Privacy &amp; Security → Full Disk Access, confirm Milady is on, then quit and relaunch Milady.

**Milady sees messages but replies never send.**
Accessibility permission is missing. Same dance: System Settings → Privacy &amp; Security → Accessibility → add Milady.app, enable it, quit + relaunch.

**Replies are green bubbles instead of blue.**
Messages.app is signed out of iMessage or the account is offline. Open Messages.app, send a manual test message — if it goes green, iMessage itself isn't working and Milady can't fix that. Sign back in via Messages → Settings → iMessage.

**Polling feels laggy.**
Lower the poll interval. 1000ms is about as low as you want to go without creating measurable CPU load on older Macs.
